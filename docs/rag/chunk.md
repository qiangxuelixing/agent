# 1. 为何分块
分块的必要性源自两个核心限制：
|限制|说明|
| :--- | :--- |
|模型上下文窗口|LLM无法一次性处理无限长度的文本，必须将长文档切分为适配模型处理能力的小文档|
|检索信噪比|若一个文本块包含过多无关信息（噪声），会稀释核心信号，导致检索器难以精确匹配用户意图|

# 2. 分块策略
## 2.1 基础分块策略
### 2.1.1 固定长度分块
- 核心思想：按固定字符数切割文本，不考虑语义结构
- 优点：实现简单
- 缺点：容易破坏句子完整性
- 适用场景：非结构化纯文本预处理，有chunk_size和chunk_overlap两个值
### 2.1.2 递归字符分块（推荐，稳妥）
- 核心思想：按优先级分隔符递归切分（\n\n -> \n -> 空格 -> ""），尽量保留段落、句子完整性。
- 适用场景：绝大多数通用文本的首选策略
### 2.1.3 基于句子分块
- 核心思想：以完整句子为单位进行组合，确保语义完整，英文文档一般是.?  中文文档建议通过正则表达式识别。！？进行切分，或使用spaCy、HanLP等支持中文的NLP工具
- 适用场景：法律文书、新闻报道、对句子完整性要求高的领域

## 2.2 结构感知分块
利用文档的固有结构（标题、列表、对话轮次）作为分块边界，逻辑更强，上下文更完整
### 2.2.1 结构化文本分块
根据文档结构进行切分，如md文档 一个#是一级标题 ##是二级标题
- 核心思想：根据标题层级（如 `#`， `##`）或HTML标签切分
- 适用场景：技术文档、博客、手册等格式规范的文档
### 2.2.2 对话式分块
- 核心思想：按发言人或对话轮次切分
- 适用场景：客服记录、访谈、会议纪要
```示例
"客户：这个商品多收50元",
"客服：您说的是0523这个单吗",
"客户：是的",
"客服：好的，你稍等我核实一下"
```
⚠️可扩展：加入发言人数统计、情感变化检测等作为切分依据

## 2.3 语义与主题分块
超越物理结构，基于内容语义或主题变化进行切分
### 2.3.1 语义分块
- 核心思想：计算相邻句子的向量相似度，在语义突变处切分
- 适用场景：知识库、论文、多话题文档

⚠️ 注意向量相似阈值，过大可能导致很难匹配相似，文档分块过多，过小可能导致相似度过高，文档分块少且块比较大
### 2.3.2 基于主题的分块
- 核心思想：是用LDA主题模型识别段落主题，在主题切换时切分
- 适用场景：长篇报告、书籍章节
> ⚠️ 注意事项
> - 主题模型对短文本标果差
> - 预处理影响大（去停用词、词干还原）
> - `n_topics` 难以预估，需反复实验
## 2.4 高级分块策略
### 2.4.1 小-大分块
- 核心思想：
  - 小块用于检索（高精度）
  - 大块用于生成（丰富上下文）
- 适用场景：复杂问答系统，需兼顾检索准确性与生成完整性
### 2.4.2 代理分块
- 核心思想：用LLM模拟人类阅读，动态识别知识单元
- 适用场景：实验性项目、高度非结构化文本

# 3. 混合分块
## 核心思想
先用宏观策略（如结构化分块）粗切，在对过大的块使用精细策略（如递归或语义分块）细切。
## 适用场景
结构复杂、内容密度不均的文档（如技术白皮书、企业年报）

# 4. 如何选择分块策略
## 分层决策框架

|步骤|策略|说明|
| :--- | :--- | :--- |
|第一步：基准策略|RecurslveCharacterTextSplitter|通用性强，效果稳定，适合建立性能基线|
|第二步：结构优化|MarkdownHeaderTextSplitter等|若文档有结构，有限使用结构化分块|
|第三步：精度瓶颈|SemarticChunker或ParentDocumentRetriever|检索不准或生成不完整时引入语义/小大分块|
|第四步：极端复杂|混合分块|多格式、长文本、高密度场景的终极方案|

# 4. 结论：分块是实践，而非理论
## 核心认知
1. 没有银弹
   没有任何一种方法适用于所有场景，必须根据数据特征、业务需求持续迭代优化。
2. 始于简单，终于复合
   从 `RecursiveCharacterTextSplitter` 建立基线，逐步引入结构化、语义化、混合策略，是最稳健的路径。
3. 分块即建模
   你怎么切文档，就等于你怎么理解知识，高质量的分块本身就是一种语义增强与结构化的过程。
> 🌟最终建议
> 把分块当成一个可度量、可实验、可优化的工程模块，结合评估指标（如检索命中率、答案准确率）进行A/B实验，才能真正发挥其价值

📌 附录：实用建议清单

- ✅ 小-大分块适合高要求问答系统
- ✅ 代理式分块尚处于实验阶段，谨慎用于生产
